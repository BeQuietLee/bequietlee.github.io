Google Architecture Sample TODO MVP 学习

2018.06.26 Li Lei  热度 ℃

学习 googlesamples/android-architecture 的第一篇文章

文章思路

我认为MVP模式之所以适合学习和推广，在于它总结出了App最常见的形式，大部分App都是按照“数据、显示、交互”的形式来设计的。在MVP模式中，Model负责维护数据，并提供存取的接口，通常有本地持久化和网络两种途径。View负责绘制界面，并给出控制界面元素变更（包括页面跳转）的接口。一切用户可以进行的页面操作，都被放入Presenter中。最后，由Activity负责将这三者组织到一起，通常Activity的代码不会很复杂。下面结合官方代码进行解读。

BaseView和BasePresenter

所有页面的Presenter都实现了BasePresenter接口，它只有一个start()方法，是页面启动后的入口，只会在Fragment（也就是View对象）的onResume()方法中被调用。而为什么BasePresenter.java被写成了一个接口而非抽象类？我认为这与项目中各个页面的接口设计有关，View和Presenter对外开放的接口都会被整合入XXXContract.java中，每个页面都如此，而接口自然是只能继承自接口的，所以BasePresenter.java只能是Interface，BaseView.java也是如此。

BaseView.java中也只有一个setPresenter(T presenter)方法，会在Presenter构造完成后（构造函数最后）进行调用。这里使用范型，是为了保障同一个页面的View只能由对应的Presenter持有。你不可以将AddEditTaskView与TaskDetailPresenter进行关连，反之亦然。

Data层

Data（数据）层的代码都位于data包中，由于todoApp比较简单，只有一种元数据Task。

Task的所有操作都位于TasksDataSource这个接口中，demo里使用回调来单独/批量取task，因为是异步请求的缘故。对于如完成、保存、删除等操作则是同步接口，这里我比较困惑，按理来说这些操作都存在失败的可能，类注释里也提了是出于简化的原因才只在加载Task的接口做了异步处理，实则应当所有接口都如是。但每个接口上都有一个Callback，不会很冗杂吗？有没有更简便的写法？

TasksLocalDataSource和TasksRemoteDataSource分别从本地和网络两个维度实现了上面的接口，然而在Data层承担对外功能的是TaskRepository这个类，它集成了Local和Remote的实现，在外部对它进行调用时，是不知道数据来源究竟是本地或者网络的。需要在构造函数里传入这两个实现对象。

具体到每一个数据操作的调用，就是将其传递给Local/Remote数据源的过程，其中Local数据源使用了android.arch.persistence.room包，好处是简化了声明表名、字段名、增删改查语句的过程，能够很好地将表中的每一行数据与Task对象自由转换。ToDoDatabase.java用于初始化数据库，TasksDao接口（不需要写实现，由room自动生成）包含了所有的增删改查操作;不需要显式建表语句，只需将表名、字段名写在Task.java中，框架会自动创建对应表。

Remote数据源则使用handler模拟延时网络操作。

TasksRepository 里用到了 Memory Cache，这里稍微深入一下。

基础服务：MemoryCache

getTasks()的实现，是逐级从cache、local、remote来取数据，直至remote取失败才进行失败回调。需要注意的是项目里在构建TasksRepository时，将FakeTasksRemoteDataSource作为remote数据源传入，其中只是维护了一个列表，用于存取。

mCacheIsDirty用于标明xxxx，通过refreshTasks()来重置它，对应页面上的操作就是在菜单里选择refresh（下拉刷新不会重置）。

当mCacheIsDirty为true时，从remote取数据，取回数据猴更新cache和local

Memory Cache在数据操作中的使用过程：

  1. getTasks: 先判断dirty，如果false则直接返回cached tasks，如果true说明本地数据和内存缓存不一致，需要从远端全量更新cache和local。只要数据来源不是cache，在拉取完成后都会全量更新local与remote。
  2. getTask: 不关心dirty，直接从memory cache里面取，若为null则先看本地，本地为null或者失败再看remote。获取成功后只更新 memory cache。
  3. saveTask: 分别调用local和remote的saveTask，并put入cache中。
  4. activateTask/completeTask: 同saveTask。
  5. deleteTask/deleteTasks: 同saveTask，只是将put换成 remove。

采用Memory Cache另一个好处是，如果项目里用到了MVVM框架，可以很好地利用Memory Cache将local和remote进行整合进而绑定UI。

基础服务：线程池/ThreadExecutor

项目里面处理异步操作主要集中在util包下AppExecutors这个类，它提供了本地存储、网络、UI变更三个异步处理器。对外开放出获取这三个处理器的接口。这是一个不错的可以参照的Executors框架。

其中disoIO对数据库进行串行操作，使用的是SingleThreadExecutor；networkIO存在并行可能，使用了count为3的FixedThreadPool；mainThread将事件抛给主线程的Looper中。

添加/编辑人物页面（AddEditTask）分析

AddEditTaskActivity流程简述

  1. 初始化Activity，设置ActionBar；
  2. 创建Fragment，将taskId（从Intent里传入）传给Fragment；
  3. 使用taskId、fragment和tasksRepository作为参数构造Presenter；
  4. 在View里调用Presenter的start方法，加载数据并更新UI。

首先看一下Presenter与View的接口，代码位于AddEditTaskContract.java。

AddEditTaskContract.View 提供了所有 UI 的显示与变化， AddEditTaskContract.Presenter 则包含用户可以进行的操作。

AddEditTaskFragment.java 实现了 AddEditTaskContract.View 的接口，控制页面上所有 UI 元素的显示，包括点击事件绑定与处理等，接口中调用 Presenter 进行数据处理，例如 saveTask 操作：

AddEditTaskPresenter.java 实现了 AddEditTaskContract.Presenter 接口，它充当连接 Data 和 View 的桥梁，在数据处理成功/失败时调用对应的 UI 方法。

在阅读代码时我产生了一个疑问 🤔 ，️Fragment 是在其 onResume 方法里调用 Presenter 的 start() 方法，作为整个页面启动入口的，而且并没有进行 null check ，如果此时 Fragment 持有的 Presenter 对象为 null ，就会发生问题，代码里却没有明确保证这种顺序。找找看 Presenter 对象是何时被传递给 Fragment （也就是 View ）的？是在 Activity 的 onCreate() 方法末尾，也就是 Fragment 被构造完成后。唯一合理的解释是 Fragment 的 onResume() 方法一定发生在其 Activity 的 onCreate() 方法之后，是系统保证了这一点。

总结

官方 Demo 中值得学习的地方还有很多，还有 Injection 、 Mockto 等用法都没有在文中涉及，文章基本代表了自己目前阶段对 MVP 的认识。较多篇幅用在data层设计与线程池的讲解，算是弥补自己之前在基础知识方面的忽视。todoMVP 早在 2016 年就已面世，现在写这篇文章不知算不算过时。但我相信小到客户端开发、大到程序设计，总有些原理性的知识是不会随时间推移而改变的，希望自己可以早日把握其中的“本质”。
